# Selected Stories

Истории из рабочего опыта в деталях — не резюмешный формат, а как это реально прожилось. Каждая — отдельный кейс с причинами, решениями и тем, что я из неё вынес.

## История 1: Casino production incident — как это было прожито

### Ситуация
Инцидент на enterprise-продукте (онлайн-казино в Восточной Европе). Сбой Kafka сломала **другая команда** в компании. Когда мы обратились к ним для восстановления, они сказали: «не будем чинить и восстанавливать, слишком трудозатратно».

Наш delivery manager согласился забрать задачу на нашу команду. Это оказалось двойной ошибкой: **забрав решение, он одновременно забрал и ответственность за все последующие инциденты на нас.** Второй проблемой стало сильное давление по срокам со стороны клиента.

### Действия
Делали спокойно, не в панике. Разделили работу на две итерации.

**Первая прогрузка:**
- Спроектировал кастомный Go-сервис для backfill: проход по Kafka backlog → агрегация сообщений → реэмит в customer-обработчики
- Из 200k+ потерянных сообщений восстановили ~180k
- Обнаружили, что 20k сообщений были с другим типом и не попали в первый фильтр

**Вторая прогрузка:**
- Расширил сервис для обработки второго типа сообщений
- Догнал оставшиеся 20k

### Результат
Весь backlog восстановлен. Ключевой клиент компании удержан. Контур стал более устойчивым к подобным сбоям.

### Что вынес
Когда delivery manager соглашается на условия другой команды в межкомандных переговорах, он не просто "берёт задачу" — он принимает на команду ответственность за все последующие инциденты по этой области. Это нужно обсуждать до согласия, а не после.

## История 2: Viralmaxing онбординг — падение Reg→Trial

### Ситуация
После планового редизайна онбординга в VM конверсия Reg→Trial начала проседать (с ~31% до ~19%). Обычный reflex — крутить гипотезы в вакууме или продавливать «правильный» дизайн.

### Действия
Вместо гипотез пошёл в PostHog session replay. Смотрел реальные сессии пользователей.

Увидел конкретную точку застревания: последний редизайн добавил шаг «введите конкурентов» как обязательный — люди останавливались именно на нём. Многие закрывали вкладку, не дойдя до trial.

Решение: сделал ввод конкурентов опциональным, упростил путь. Релиз пошёл в текущий спринт.

### Результат
Фикс в работе. Ожидание — поднять Reg→Trial обратно к 25%+.

### Что вынес
Session replay — не опциональная штука, а обязательный инструмент между метрикой и решением. Метрика показывает «что упало». Replay — «почему упало». Докручивать гипотезу без replay = гадание по кофейной гуще.

## История 3: HikeMySkill — запустил и закрыл

### Ситуация
2024 — запустил свой EdTech стартап (HikeMySkill) как соло-фаундер. Тяжело вкладывался в код: писал много, почти не разговаривал с юзерами. Начал строить enterprise-систему под B2B аудиторию.

Затем вышел GPT-4. Рынок образования начал быстро меняться — большая часть value propositions EdTech-платформ оказалась легко воспроизводима через чистую модель.

### Действия
Оценил рынок трезво: bet'ить EdTech-продуктом против прямого доступа к большим моделям — проигрышная стратегия. Дальнейшие инвестиции времени не окупятся.

Закрыл продукт. Не тянул месяцами, не давал умирать в фоне — свернул быстро.

Следующий заход (Viralmaxing) начал с противоположного паттерна: разговоры с юзерами первыми, код потом.

### Результат
Proof of shipping discipline (запустил от 0 до прода соло). Proof of умения честно закрывать, когда рынок изменился. Применил полученный опыт к следующему продукту.

### Что вынес
**Урок 1:** рынок побеждает идею. Если появилась технология, которая делает value proposition продукта тривиальным — дальше биться бесполезно.

**Урок 2:** «закрыт в коде» = невидимый юзер = продукт не вырастет. С тех пор в VM общаюсь с юзерами каждую неделю, без исключений.

## История 4: Viralmaxing — B2B→B2C гипотеза, которая пока не закрыта

### Ситуация
У Viralmaxing работает B2B трек: agency-модель, profitable с октября 2025. Мы решили расшириться через B2C — запустить SaaS с PLG-воронкой на массовых креаторов.

Собрал: отдельный онбординг под B2C, подписочную монетизацию, PostHog-аналитику, content distribution.

### Действия
- Запустил SaaS в январе 2026, zero paid acquisition
- Рост по B2C-треку есть, но несравнимый с agency

По динамике видно: agency-трек сильнее по результатам. Гипотеза PLG-scale'а пока **не подтверждена** на нужном уровне.

### Результат
Гипотеза открыта. Решение в Q3 2026: либо другой B2C ICP (сужение сегмента), либо возврат фокуса на расширение B2B. Сейчас тестирую ниши и итерирую онбординг.

### Что вынес
«Расширение в новый сегмент» — это не просто «запуск другого онбординга». Это другой продукт с другой воронкой, другим ICP, другим unit economics. Недооценил разницу. Не скрываю, что гипотеза ещё открыта — founder-зрелость в признании этого открыто, а не в подкрашивании цифр.

## История 5: Trade-off «быстрый модуль vs core архитектура»

### Ситуация
В VM регулярно появляются фичи-кандидаты: что-то проверить на клиентах, увидим — важно оно или нет. Если каждую такую гипотезу строить «правильно» в core — каждая проверка занимает недели, большинство из них disposable, время теряется.

### Действия
Принципиальный паттерн:
1. **Гипотеза идёт в отдельный быстрый модуль.** Не в core. Не в красивую архитектуру. Быстрая реализация, отдельный код, понимаю что он может быть выброшен.
2. **Клиенты подхватили → другие фичи начали её использовать** → перелопачиваю в core, доплачиваю архитектурой.
3. **Гипотеза умерла → удаляю.** Не потратил неделю на «правильную» архитектуру для кода, который удаляется.

### Результат
Высокая пропускная способность по гипотезам при сохранении чистоты core. Текущий цикл «идея → прод» для быстрого модуля — дни.

### Что вынес
Это **не техдолг**. Техдолг — это когда делаешь плохо без осознания. Здесь — **осознанное откладывание инвестиций** до момента, когда они окупятся.

Скорость > качество, пока гипотеза не закрыта. Качество > скорость, когда гипотеза подтвердилась. Это два разных режима работы с одним и тем же кодом.

## История 6: AI plan-mode iteration — мой реальный flow с Claude Code

### Ситуация
Многие разработчики при работе с AI либо сливают (принимают первое предложение), либо игнорируют (не доверяют и пишут сами). Оба режима — потеря.

### Действия
Мой flow с любой нетривиальной задачей в Claude Code / Codex:

1. **Plan mode** первым шагом. Не пишу код сразу.
2. AI генерирует первое предложение архитектуры / подхода.
3. **Первое предложение почти всегда плохое:** переусложнено, игнорирует существующий код, выбирает не тот паттерн, добавляет зависимости без причины.
4. **Докручиваю в plan mode итеративно:** «убери X», «переиспользуй Y из pkg/», «упрости до Z».
5. Когда план выглядит чисто — только тогда переходим к реализации.

### Результат
Докручивание плана занимает 5-10 минут. Сэкономленная реализация — часы и иногда дни. Получаемый код — в стиле проекта, без переусложнения.

### Что вынес
«AI делает за меня» — миф, который ведёт к хаосу в коде. AI ускоряет, если ты им управляешь. AI сливает качество, если ты ему сдаёшься.

**Модель AI-flow:** AI = junior engineer с потенциалом. Мой plan mode = code review перед кодом. Без моего reviewer'а модель не справляется. Я справляюсь в 10 раз быстрее с моделью.

Параллельно работают ресёрч-агенты для постов и Zettelkasten-агент для обработки видео-транскриптов. Оркестрация — моя ответственность.
